# 기회-해결책 트리(OST) 프레임워크 가이드

출처: Teresa Torres의 Continuous Discovery Habits 기반

## 개요

OST(Opportunity-Solution Tree)는 비즈니스 아웃컴에서 출발하여,
기회(충족되지 않은 사용자 니즈)를 계층화하고 그에 맞는 해결책을 연결하는 전략 구조.
"무엇을 만들 것인가"보다 "왜 만드는가"를 명확히 하기 위해 사용한다.

---

## 1. OST 3계층 구조

### 계층 1: 비즈니스 아웃컴 (Outcome)
- **정의**: 우리가 이 디스커버리를 통해 달성하려는 비즈니스 목표
- **표현 방식**: 측정 가능한 지표로 표현
- **예시**: "리셀 거래 성사율 30% 개선"

### 계층 2: 기회 (Opportunity)
- **정의**: 아웃컴 달성을 방해하는 충족되지 않은 사용자 니즈 또는 Pain Point
- **표현 방식**: 사용자 관점의 니즈 또는 문제 상황
- **JTBD와 연결**: 각 기회는 하나 이상의 JTBD와 매핑되어야 함
- **예시**: "구매자가 거래 전 스타일 적합성을 빠르게 판단하지 못함"

### 계층 3: 해결책 (Solution)
- **정의**: 기회를 해소하기 위한 구체적 접근 방식
- **표현 방식**: 기능 또는 경험 수준 (구현 방법 아님)
- **복수 해결책**: 하나의 기회에 여러 해결책 후보를 나열하고 평가
- **예시**: "스타일 태그 기반 필터링 기능" / "셀러 스타일 무드보드 프로필"

---

## 2. OST 작성 절차

### Step 1: 아웃컴 고정
먼저 하나의 비즈니스 아웃컴만 선택한다 (복수 아웃컴은 별도 OST 작성).

### Step 2: 기회 트리 구성
인터뷰와 시장 분석에서 도출된 Pain Points와 JTBD를 아웃컴에 연결한다.

**기회 계층화 예시:**
```
아웃컴: 리셀 거래 성사율 30% 개선
    └─ 기회 A: 구매 결정 과정에서 신뢰 부족
        └─ 기회 A-1: 판매자 신뢰도 판단 어려움
        └─ 기회 A-2: 상품 상태 정보 부족
    └─ 기회 B: 스타일 매칭 정확도 낮음
        └─ 기회 B-1: 스타일 취향 기반 검색 불가
        └─ 기회 B-2: 스타일 맥락 없이 단품 사진만 제공
```

### Step 3: 해결책 매핑
각 기회(특히 하위 기회)에 2~3개의 해결책 후보를 연결한다.

### Step 4: 해결책 평가
아래 기준으로 우선순위를 결정한다:

| 평가 기준 | 점수 (1~5) |
|---------|----------|
| 기회 크기 (얼마나 많은 사용자가 겪는가) | |
| 실현 가능성 (기술적/시간적/비용적) | |
| 인터뷰 수용도 (사용자가 실제로 원하는가) | |
| 경쟁사 차별화 (경쟁사가 없거나 약한가) | |

**최종 점수 = 평균값 → 가장 높은 해결책을 1순위 권장**

## 3. OST Markdown 작성 형식

```markdown
## 기회-해결책 트리 (OST)

**비즈니스 아웃컴**: {지표와 목표 수치}

---

### 기회 #1: {기회 설명}
> 연결 JTBD: JTBD-001 | 연결 Pain: PC-01, PC-02 | 기회 크기: 상/중/하

#### 기회 #1-1: {하위 기회 설명 (선택)}

**해결책 후보:**

| 해결책 | 기회 크기 | 실현 가능성 | 수용도 | 차별화 | 종합 |
|--------|---------|-----------|--------|--------|-----|
| 1-A: {솔루션} | 4 | 3 | 5 | 4 | **4.0** |
| 1-B: {솔루션} | 3 | 5 | 3 | 2 | **3.25** |

> 권장: **1-A** — {근거 한 줄}

---

### 기회 #2: ...
```

## 4. 흔한 OST 실수

| 실수 | 올바른 방향 |
|------|-----------|
| 아웃컴을 기능으로 작성 | 아웃컴은 비즈니스 지표여야 함 |
| 기회를 해결책으로 작성 ("추천 기능 부재") | 기회는 사용자 니즈 ("원하는 스타일을 빠르게 찾지 못함") |
| 단 하나의 해결책만 연결 | 기회당 최소 2개 해결책 후보 필요 (비교 없이는 선택 없음) |
| 기회 간 중복 | 각 기회는 서로 다른 사용자 니즈를 반영해야 함 |
| 데이터 없이 해결책 평가 | 인터뷰 수용도를 반드시 포함 |
